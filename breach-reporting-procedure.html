Rapix Connect - Information Security Breach Reporting Procedure

(Pilot Phase & Future Operations)

1. Purpose

This procedure outlines the steps for Rapix Connect co-founders to manage suspected or confirmed information security breaches, covering the current pilot and future operations. It aims for timely, effective breach management, minimizing harm, and ensuring compliance with UK data protection laws (UK GDPR, Data Protection Act 2018).

This procedure acknowledges Rapix Connect's roles:

  • As a Data Processor for UK NHS Electronic Prescription Service (EPS) codes processed for client pharmacists/GPs.
  • As a Data Controller for user relationship data, including Supabase User IDs stored in its own database, and for the processing of user authentication data (like NHSmail addresses) handled via Supabase.

2. Scope

This procedure applies to any incident involving potential or actual unauthorized access, loss, destruction, alteration, or disclosure of:

  • Personal data for which Rapix Connect is a Data Controller (e.g., Supabase User IDs in Rapix Connect's database; indirectly, NHSmail addresses of GP users managed via Supabase through Rapix Connect's account).
  • Personal data for which Rapix Connect acts (or will act) as a Data Processor (NHS EPS codes).
  • Credentials providing access to third-party services storing personal data on behalf of Rapix Connect (e.g., Supabase).
  • Other sensitive or confidential information.

3. Definitions

  • Personal Data Breach (UK GDPR): A security breach leading to accidental or unlawful destruction, loss, alteration, unauthorized disclosure of, or access to, personal data.
  • Data Controller (Rapix Connect for):
    • Supabase User IDs stored in its application database.
    • The purposes and means of processing GP user NHSmail addresses (and other authentication data) managed through its Supabase instance.
    • Potentially names of sole trader pharmacies.
  • Data Controller (Client Pharmacists/GPs for): Patient data linked to EPS codes.
  • Data Processor (Rapix Connect as): For EPS codes processed for client pharmacists/GPs.
  • Data Processor (Supabase as): For user authentication data (e.g., NHSmail addresses, OTP handling) processed on behalf of Rapix Connect.
  • EPS Codes: Electronic Prescription Service codes. Encrypted at rest and in transit by Rapix Connect; patients not identifiable by Rapix Connect from codes alone.
  • Supabase User ID: An identifier stored by Rapix Connect, linking to user accounts managed in Supabase. Pseudonymous personal data.
  • ICO: Information Commissioner's Office.
  • Data Processing Agreement (DPA): Legally binding contract between a Data Controller and Data Processor.

4. Roles and Responsibilities

Both Co-founders are jointly responsible for:

  • Understanding and adhering to this procedure.
  • Immediate internal reporting of suspected breaches.
  • Collaboration on breach response.
  • Maintaining breach records.
  • Ensuring compliant DPAs are in place with all clients (for EPS data processing) before official service launch AND with essential data processors like Supabase.

Software Engineer Co-founder (Technical Lead):

  • Leads technical investigation, containment, eradication, recovery.
  • Manages security of Rapix Connect systems and credentials for third-party services like Supabase.
  • Advises on technical aspects.

Pharmacist Co-founder (CEO - Compliance & Communications Lead):

  • Leads overall breach management and risk assessment.
  • Manages communications with affected Data Controllers (clients) for EPS data breaches.
  • Manages communications with ICO and affected GP users if a breach of data controlled by Rapix Connect occurs (including via Supabase compromise originating from Rapix Connect's end).
  • Ensures reporting obligations are met.
  • Documents the breach response.

5. Reporting and Response Procedure

Step 1: Discovery and Initial Internal Reporting

  • Immediate communication between co-founders of any suspected incident.
  • Initial details: What, when, who, systems/data (including potential Supabase impact), actions?
  • Monitor user reporting channels (hello@rapixconnect.com, public phone number) and notifications from processors like Supabase.

Step 2: Immediate Actions & Containment (Technical Lead)

  • Implement actions to contain the breach:
    • Isolating systems.
    • If Supabase access is suspected to be compromised: Immediately take steps to secure the Supabase account (e.g., change passwords, review API keys, contact Supabase support if needed).
    • Changing credentials for Rapix Connect systems.
    • Preserving evidence.

Step 3: Assessment and Triage (Both Co-founders)

  • Confirm if a personal data breach occurred.
  • Identify the type(s) of data and systems involved:
    • Rapix Connect database (Supabase User IDs)?
    • Rapix Connect's access to its Supabase instance (potentially exposing NHSmail addresses, etc.)?
    • EPS codes (Rapix Connect as Processor)?
  • Assess severity and scope: Consider if the breach is of Rapix Connectâ€™s own database or a compromise of access to data held by Supabase.
  • Assess risk:
    • Breach of Supabase User IDs from Rapix Connect DB: Assess risk. While pseudonymous, is linkage possible or harm likely?
    • Breach of Rapix Connect's Supabase Credentials/Access (exposing NHSmail addresses, etc.): This is likely to be high risk. Determine if ICO/data subject notification is required.
    • For EPS Code Data (Processor): Notify the Data Controller (client).
  • Document assessment.

Step 4: Notification (Communications Lead, based on assessment)

  • Internal Record Keeping: Document ALL personal data breaches.
  • Processor Notifications (e.g., from Supabase to Rapix Connect): If Supabase notifies Rapix Connect of a breach on their end affecting your data, activate this response procedure.

If the breach involves EPS Codes (Rapix Connect as Processor):

  • Notify affected Data Controller(s) (Clients) without undue delay.
    • Pilot: Inform pilot participants.
    • Official Operations: Follow DPA terms.
  • Assist Data Controllers.

If the breach involves data controlled by Rapix Connect (e.g., Supabase User IDs from own DB, or NHSmail addresses via compromised Supabase access from Rapix Connect's end):

  • Information Commissioner's Office (ICO):
    • Notify ICO without undue delay (ideally within 72 hours) if the breach is likely to result in a risk to individuals' rights and freedoms. A compromise of credentials leading to access to NHSmail addresses would likely meet this threshold.
  • Affected Data Subjects (GP Users):
    • Inform affected GP users directly and without undue delay if the breach is likely to result in a high risk (e.g., their NHSmail and authentication details exposed via Supabase due to Rapix Connect's account compromise).

Step 5: Investigation and Recovery (Technical Lead, Communications Lead supports)

  • Detailed root cause analysis.
  • Recover systems/data, implement preventative measures (e.g., review Supabase security settings, API key rotation, access controls).

Step 6: Post-Incident Review and Lessons Learned (Both Co-founders)

  • Thorough review.
  • Update procedures, policies, DPAs, security controls (including for third-party services).
  • Document.

6. Key Considerations & Best Practices

  • Data Processing Agreements (DPAs): CRITICAL.
    • With Clients (Pharmacists/GPs): Essential before official launch for EPS data.
    • With Processors (Supabase): Ensure a valid DPA is in place with Supabase. Review their terms regarding security and breach notification.
  • Security of Credentials: Implement strong security measures for all credentials, especially those providing administrative access to services like Supabase (e.g., strong unique passwords, multi-factor authentication where available for admin accounts, secure API key management).
  • Understanding Data Flows: Clearly map where all personal data resides (Rapix Connect DB, Supabase, etc.) and who is responsible at each stage.
  • Legal Counsel: Strongly recommended for DPA drafting/review and overall GDPR compliance.
  • Data Security and Protection Toolkit (DSPT): Investigate relevance for official NHS operations.
